Basic Priciples

This documentation is the most comprehensive guide to penguin.js. All topics (except for components) are covered here. First we start with the idea behind important basic priciples:

Environments

A penguin site needs two environment servers. CMS server Dynamic, and hosts the CMS. ; Production server Static, and all published websites are compiled into static html and pushed into this simple file server. Thus, the production server acts more like a CDN, whereas the CMS/Editing server acts like a backend. We then change our files on the CMS server and deploy it to the production server.

Pages and Objects

Filesystem as an API True to the foundational Linux concept of "Everything is a file", penguin.js acquires its power by embracing the filesystem as its API. This allows it to abstract from any single database or storage system, giving the user the flexibility to use whatever suits him best. All the program needs to know is an interface with which to connect to it - particularly how to read and write. There are two types of websites: pages and objects. pages are simple websites. objects are reusable templates which can be used for different contents.

Components

Modular "components" give penguin.js the flexibility and power to adapt. Essentially, each component gives your (html) element a specific functionality that is not dependent on other components. This means that you can mix and match them until you get your desired website. For example, there are components for inplace editing of texts and images, which you can render on a specific HTML tag. Furthermore, you can install components via npm or even create them by yourself.

Structure of package.json

One of the central elements of penguin.js is the package.json since we use it for cofiguration.

{
  "penguin": {
    "languages": ["en"]
  },
  "components": {
    "Inplace": "penguin-inplace",
    "SaveButton": "penguin-save-button",
    "SavedIndicator": "penguin-saved-indicator"
  }
}

penguin is the root of your configuration. Keys inside:

  • languages: This is a mandatory field. It lists all languages of the project. The first language is the default language. Note that this must have at least one element.
  • components: This is an object that contains the components of your project
  • globals: This is an array that contains all global fields in your project. A field may either be a string or an array. It is an array if the field is not localized. Below are examples of both forms:
    "site-title"  and ["site-title", {"localized" :false }]

A quick note on globals

Penguin.js comes with rich support managing global fields (values that are the same over the whole project). In addition you can have fields that are not localized (the same over all languages). That is e.g. useful for images.

The folders

As mentioned above, penguin.js is easy because it embraces the filesystem as its API. These are the folders that have a certain function.

/pages
/objects
/static
/files
/components

/pages

This are your normal websites. If you create a file called pages/whatever.html and add some content, it will be served under /en/whatever.

/objects

This are templates which the user can save many times with different contents. Let's say you have a template for your projects. Every project has the same layout, just with different contents. If you create a file called objects/projects.html and add some content, it will be served under /en/projects/whatever. This means, you can create objects by just entering a new project slug at the end of the url. Objects can be created by user by saving. Think about it as ///.

/static

Static files like images, js, css go here. Is served under /static.

/files

Technical files that must reside under root level (website.tld/something) go here. We use it e.g. for the CNAME file when deploying to GithubPages.

/components

All project related components you qrite yourself go here. You can find out more about writing your own components in the component documentation.
